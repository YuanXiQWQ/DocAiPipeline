# DocAI Pipeline — 项目流程与任务规划

> 基于客户邮件（2026-03-03 ~ 2026-03-04）整理的完整业务流程与系统实现计划。

---

## 一、业务背景

**客户公司**：Terra Drvo d.o.o.（塞尔维亚 Ljukovo），从事欧洲橡木（欧橡）原木采购、加工（刨切、锯切）、表板生产与销售。

**核心痛点**：大量纸质单据需要人工录入 Excel，耗时且易出错。希望通过扫描件自动识别 → 自动填入 Excel 模板。

---

## 二、客户业务流程（原木全生命周期）

### 阶段 1：进口采购

**触发**：从克罗地亚（Hrvatske Šume 林业局）或其他供应商采购原木。

**报关公司**发来一组文件（每批一组）：

1. **CR_*.pdf** — 进口税款缴纳凭证
2. **Deklaracija_*.pdf** — 报关单（JCI 海关申报单）
3. **Racun.pdf** — 商业发票（供应商开具）
4. **SKEN.*.pdf** — 扫描件合集（多页 PDF，包含商业发票、EUR.1 产地证、检疫证书、CMR 运输单、运费发票等）

**目标输出**：将这些文件中的关键信息抽取后填入 → **`纸质发票记录表`** Excel

| 纸质发票记录表字段  | 来源文件                    |
|------------|-------------------------|
| 所属人、公司名    | 固定值（根据业务配置）             |
| 供应商名称      | 商业发票(Racun) / SKEN 中的发票 |
| 性质（进口/国内）  | 报关单(Deklaracija)        |
| 名称（原木/运费等） | 商业发票 / 运费发票             |
| 日期         | 发票日期                    |
| 发票号        | 发票编号                    |
| 编号         | 内部编号（可能需要规则生成）          |
| FSC（是/非）   | EUR.1 证书或发票备注           |
| 供应商/备注     | 供应商名称                   |
| 外币金额       | 发票总金额                   |
| 数量（m³）     | 发票中的体积数量                |

### 阶段 2：原木入库 — 检尺

**触发**：原木到货后，库管人员手工检尺。

**涉及文件**：

- **克罗地亚Ad公司原木检尺及入库单-库管填报2.pdf** — 手写检尺单（Ad公司供应的原木）
- **克罗地亚Hrvatske原木对方码单及检尺入库单.pdf** — 包含两部分：
    - 第1部分：供应商（Hrvatske Šume）出库码单（对方提供）
    - 第2部分：我方库管的检尺入库单

**检尺数据**：编号、长度（米）、径级/直径（厘米）→ 计算体积（m³）

**客户期望操作**：

1. 将供应商出库码单扫描件 → 转为 Excel → 提取编号、长、直径、m³ → 填入 `原木出入库` 表的对应位置
2. 将我方检尺入库单扫描件 → 转为 Excel → 用编号做 VLOOKUP 与供应商数据核对
3. **由库管将供应商的数量与我方检尺数量填到表3（验收入库单）中**

**目标输出**：→ **`原木出入库-3-3.xlsx`** 的 `数据源表` sheet

| 数据源表关键字段 | 来源        |
|----------|-----------|
| 日期       | 入库日期      |
| 车间       | 大锯车间（固定）  |
| 工序       | 来料        |
| 木种       | 欧橡        |
| 货物名称     | 原木        |
| 序号/批次号   | 检尺单批次号    |
| 送货客户     | 供应商名称     |
| 车牌号      | CMR 运输单   |
| 包号/根号    | 每根原木编号    |
| 等级       | F1/F2/厚皮等 |
| 长度       | 检尺数据      |
| 直径       | 检尺数据      |
| 数量 m³    | 根据长×直径计算  |

### 阶段 3：原木出库/领用

**触发**：大锯车间从仓库领用原木进行加工。

**涉及文件**：

- **原木领用出库表.pdf** — 手写的领用出库表

**客户期望**：扫描件自动转为 Excel → 填入 `原木出入库` 表（工序=出库）

### 阶段 4：刨切加工

**触发**：原木在大锯车间领用后，切成大方，送入蒸煮池浸泡 3-5 天，然后上刨切机。

**涉及文件**：

- **刨切木方入池表.pdf** — 手写入池记录
- **刨切木方上机表.pdf** — 手写上机记录

**客户期望**：扫描件 → Excel → 填入 **`刨切木方入池与上机表3-3.xlsx`**

**业务说明**：

- 入池表和上机表是一线员工每天手写统计，交给库管录入 Excel
- 刨切生产 1.2MM 和 2.0MM 厚度的表板
- 表板数量到下一工序（剪切）打包后才统计

### 阶段 5：表板打包与销售

**触发**：刨切产出的表板经剪切打包后成为成品。

**涉及文件**：

- **表板打包报表.pdf** — 手写的打包报表（一线人员每天报给库管）

**客户期望**：扫描件 → Excel → 填入 **`表板统计 - 20260304.xlsx`**

### 阶段 6：整体报表汇总

**最终汇总表**：**`新公司LKV表板统计.xlsx`**

- 所有前面各环节的 Excel 数据最终汇入这个整体报表
- 通过**数据透视表**查看库存的进销存情况
- 这是管理层查看的核心报表

## 四、系统模块划分

### 模块 A：进口单据识别与录入 ✅

- **输入**：报关公司发来的 PDF 组（CR, Deklaracija, Racun, SKEN）
- **处理**：VLM 抽取 → 字段映射 → 填入纸质发票记录表
- **关键实现**：多批次关单号匹配、importer/exporter 反转纠正、欧洲数字解析、供应商映射
- **当前状态**：✅ 已完成，3批次7文件→7有效行全部正确填充

### 模块 B：原木检尺单识别与录入 ✅

- **输入**：手写检尺单扫描件（编号、长度、径级）
- **处理**：VLM 表格模式抽取 → 计算体积 → 填入原木出入库表
- **关键实现**：LogExtractor + LogFiller，手写识别准确率 97%+，打印 100%
- **当前状态**：✅ 已完成

### 模块 C：原木出库/领用单识别 ✅

- **输入**：手写领用出库表扫描件
- **处理**：FactoryExtractor(log_output) → LogOutputFiller → 原木出入库表（工序=出库）
- **特点**：只有编号+径级无长度，需靠入库数据 VLOOKUP 匹配
- **当前状态**：✅ 已完成

### 模块 D：刨切入池/上机表识别 ✅

- **输入**：手写入池表、上机表扫描件
- **处理**：FactoryExtractor(soak_pool/slicing) → SoakPoolFiller/SlicingFiller
- **特点**：上机表是旋转90°横向表格，VLM prompt 有专门处理
- **当前状态**：✅ 已完成

### 模块 E：表板打包报表识别 ✅

- **输入**：手写打包报表扫描件
- **处理**：FactoryExtractor(packing) → PackingFiller → 表板统计表
- **当前状态**：✅ 已完成

### 模块 F：Excel 自动填充引擎 ✅

- **功能**：读取 Excel 模板 → 自动填入对应单元格 → 保留公式/格式/数据透视表
- **6 个 Filler**：InvoiceFiller, LogFiller, LogOutputFiller, SoakPoolFiller, SlicingFiller, PackingFiller
- **技术**：openpyxl + monkey-patch read_worksheets 跳过 pivot
- **当前状态**：✅ 已完成

### 模块 G：Web 前端（上传 + 复核 + 导出）✅

- **功能**：上传 PDF/图片 → VLM 自动分类 → 结构化识别 → 人工复核 → Excel 自动填充 → 下载
- **技术**：React 19 + TypeScript + Vite + TailwindCSS v4 + Lucide Icons
- **功能细节**：拖放高亮、图片预览、表格数据预览、设置页面（API Key/模型/Base URL）
- **当前状态**：✅ 已完成

### 模块 H：整体报表汇总 ✅

- **功能**：Web 端数据汇总看板（进销存概览）+ 历史数据查询
- **实现**：`/api/summary` 聚合历史记录，`DashboardPanel` 展示进口/入库/出库/加工/打包各环节统计
- **当前状态**：✅ 已完成（客户的 LKV 公式不受影响，Web 看板为附加功能）

| VLM 模型   | gpt-4.1-mini（用户可在设置页面切换）          | 成本低、多语种好、可切换     |

## 八、客户使用场景（来自客户邮件 2026-03-03 ~ 04）

> 以下是客户描述的实际工作流程，是系统设计的核心依据。

### 场景 1：进口单据录入

**操作者**：库管/财务

1. 报关公司通过邮件发来一组 PDF（CR 税款 + Deklaracija 报关单 + Racun 发票 + SKEN 扫描合集）
2. 操作者将这些 PDF 上传到系统
3. 系统自动识别各文件类型，提取关键字段（供应商、金额、数量、日期等）
4. 操作者复核识别结果，确认无误后点击"填充 Excel"
5. 系统自动将数据填入「纸质发票记录表」，操作者下载填好的 Excel

**要点**：

- 每批进口有 3-4 个 PDF，不同批次的文件命名含批次号（如 C4_3141）
- SKEN 是多页 PDF 合集，内含商业发票、EUR.1、检疫证书、CMR 等多种文档
- 系统需要自动区分 SKEN 中的各种文档类型，路由到不同的填充逻辑

### 场景 2：原木入库检尺

**操作者**：库管

1. 原木到货后，库管手工在纸质检尺单上记录每根原木的编号、长度、径级
2. 同时会收到供应商提供的出库码单（打印表格）
3. 将检尺单和码单扫描/拍照后上传系统
4. 系统 OCR 识别手写/打印数据，自动计算体积（m³）
5. 操作者复核后，系统填入「原木出入库」表
6. 库管手工将供应商数量与我方检尺数量填到表3（验收入库单）中做核对

**要点**：

- 手写字体质量参差不齐，数字识别是核心难点
- 检尺数据量大（单张可能 50-100+ 条记录）
- 需要与供应商码单做 VLOOKUP 核对

### 场景 3：工厂内部日常单据

**操作者**：一线员工填写 → 库管录入

1. 原木出库/领用：大锯车间从仓库领用原木，填手写领用出库表
2. 刨切入池：切好的大方送入蒸煮池，填手写入池记录
3. 刨切上机：浸泡后上刨切机，填手写上机产量日报表
4. 表板打包：刨切产出的表板剪切打包，填手写打包记录表

上述四种表都是**一线员工每天手写**，交给**库管扫描后上传系统识别并录入 Excel**。

### 场景 4：管理层查看报表

- 所有前述 Excel 数据汇入「新公司 LKV 表板统计」整体报表
- 管理层通过数据透视表查看进销存情况
- 此环节可能不需要自动化（客户已有公式联动）

### 10.2 故意偏离（保留，附理由）

#### D1. Java + Spring Boot → Python + FastAPI

**原因**：项目的核心工作量在 AI 模型调用（OpenAI API、图像处理、Excel 操作），Python 生态在这些领域有绝对优势（openai
SDK、OpenCV、openpyxl、PyMuPDF 均为 Python 原生库）。使用 Java 会引入大量跨语言调用开销和胶水代码，而 FastAPI 性能足够且开发效率远高于
Spring Boot。

**结论**：保留。这是经过评估的合理技术选型。

#### D2. PaddleOCR → VLM 端到端

**原因**：在实际测试中，GPT-4.1-mini 的 VLM 能力对手写体的识别准确率达到 **97%+**（打印体 100%），远超 PaddleOCR
在脏文档/手写场景下的表现。且 VLM 能同时完成 OCR + 语义理解 + 噪声过滤，无需维护独立的 HTR 管线。PaddleOCR
的优势在离线/低成本场景，而本项目用户自带 API Key，VLM 调用成本可接受。

**结论**：保留。VLM 端到端方案在准确率和工程复杂度上均优于 PaddleOCR 分支方案。

#### D3. LayoutLMv3 → VLM 端到端

**原因**：LayoutLMv3 需要大量标注数据进行微调才能适配特定表格布局，而 VLM 通过精心设计的 prompt 即可直接产出结构化
JSON，zero-shot 效果已满足需求。且 LayoutLMv3 无法处理手写体，仍需搭配 OCR，增加管线复杂度。

**结论**：保留。VLM 在 KIE 任务上的 zero-shot 能力使 LayoutLMv3 成为不必要的复杂度。

#### D4. 印刷/手写路由选择器 → VLM 文档类型分类

**原因**：提案中的 Router 是区分印刷体/手写体以选择不同 OCR 引擎。实际实现中 VLM 统一处理印刷和手写，无需区分。取而代之的是*
*文档类型分类器**（`POST /api/classify`），根据文档内容路由到不同的抽取逻辑（进口单据/检尺单/入池表等），这是更高层次的路由，功能上是提案思路的升级版。

**结论**：保留。功能等价且更实用。

#### D5. 项目范围扩展

**原因**：在与客户深入沟通后，发现报关单只是原木全生命周期中的一环。客户有 6 种不同单据需要自动化处理。扩展范围能更好地验证系统的通用性和实际价值。

**结论**：保留。正面偏离，增强了项目的完整性和实用性。

### 10.3 已纠正的偏离

#### D6. YOLO 单据检测 — ✅ 已纠正

**提案描述**：采用 YOLO 检测每张报关单边界，从大图中裁切出多张单据。

**纠正内容**：重写了 `detector.py` 的轮廓检测回退方案，现在能真正从合并扫描中检测多个文档区域。处理流程：灰度化 →
自适应阈值 → 形态学闭运算 → 轮廓检测 → 面积过滤 → NMS。发现 ≥2 个区域时分割，否则返回整页。YOLO 微调模型路径仍然保留，未来可无缝替换。

#### D7. 图像预处理管线 — ✅ 已纠正

**提案描述**：倾斜/畸变校正、去噪、对比度增强。

**纠正内容**：`preprocessor.py` 中的完整预处理流程（去噪 → 纠偏 → 对比度增强 → 锐化）已在所有处理路径中启用。
`_file_to_images()` 函数现在默认开启预处理，通过 `preprocess` 参数可控。

#### D8. 局部截图与字段对齐证据 — ✅ 已纠正

**提案描述**：保存局部截图与字段对齐证据，便于人工复核。

**纠正内容**：

- 后端：所有处理路径现在均保存页面截图到 `output/crops/`，并在 API 响应中附带 `crop_image_path`
- 后端：新增 `GET /api/crop/{filename}` 端点获取裁切图片
- 前端：复核界面每条记录上方展示原始文档图像，可点击放大，与结构化表格并列展示

#### D10. 噪声抑制 — ✅ 已纠正

**提案描述**：干笔划线、漏油点状噪声、涂改检测。

**纠正内容**：所有 VLM prompt（vlm_extractor、log_extractor、factory_extractor 共 5 个 prompt）均已添加 NOISE SUPPRESSION
规则，覆盖干笔划线、墨水点噪声、可疑小数点、扫描仪线条、涂改重写、污渍阴影等场景。

#### D11. 准确率评估 — ✅ 已纠正

**提案描述**：核心字段抽取准确率达到 80%。

**纠正内容**：新增 `evaluate.py` 评估脚本，支持精确匹配、部分匹配（LCS）、按字段/按文档类型的分类统计、提案目标自动判定。配套
`ground_truth_example.json` 标注模板。

### 10.3.1 尚未解决的偏离

#### D9. 主动学习与一键纠错 — 降级为"未来工作"

**提案描述**：主动学习：一键纠错并反向喂入少量样本用于持续改进。

**现状**：未实现。这是最后一个未解决的偏离。

**降级理由**：本项目使用 OpenAI API 调用模式，无法直接微调 GPT 模型。实现真正的主动学习需要：

1. OpenAI Fine-tuning API（仅支持文本，不支持视觉模型微调）
2. 切换到开源视觉模型（如 Qwen-VL）并本地训练
3. 两者均超出本项目范围

作为替代，用户复核时修正的数据会保存到导出的 Excel 中，未来可作为微调数据集的基础。
